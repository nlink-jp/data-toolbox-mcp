# v0.4.0 開発計画

> Status: Draft
> Date: 2026-06-06
> 前提 ADR: ADR-0010 (UX polish: describe_workspace + query_data 拡張 + delete_workspace dry_run + table-not-found hint + description hints)

v0.3.0 までで「コアの実用ループ」(load → execute → load_from_work → attach_files) は閉じた。v0.4.0 は LLM 自身が「あれば便利」とした残り 5 項目をまとめて対応する UX polish リリース。

## 1. Goals (v0.4.0 完成条件)

- 9 MCP ツール: 既存 8 + `describe_workspace`
- `query_data` 戻り値に `truncated` + `total` が乗る (truncate 時のみ追加 COUNT 実行)
- `query_data` の table-not-found エラーに `missing_table` / `available_tables_in_this_workspace` / `other_workspaces` の details が乗る
- `delete_workspace` に `dry_run` 引数追加、true なら何も消さず削除予定情報を返す
- 4 ツール (`load_data` / `execute_code` / `query_data` / `delete_workspace`) の inputSchema description に 1 行ずつ hint
- ユニット + E2E テストすべて pass
- README × 2 / AGENTS / architecture / CHANGELOG / samples が新挙動を反映
- v0.4.0 タグ + 5 platform release + darwin notarize

#### 2b. query_data 戻り値拡張 (item 2)

- `QueryDataResult` 構造体に追加:
  ```go
  Truncated             bool   `json:"truncated"`
  Total                 *int   `json:"total"`                                     // *int で null 表現
  TotalUnavailableReason string `json:"total_unavailable_reason,omitempty"`
  ```
- `LimitReached` は `Truncated` と同値で残す (後方互換)
- truncate 時 (rows > limit) のみ 2 つ目の query `SELECT COUNT(*) FROM (user_sql) sub` を execScript で別途実行
- COUNT() の timeout は execScript の既存の context.WithTimeout 経由、エラー時は Total=nil + Reason="count_timed_out"

#### 2c. delete_workspace dry_run (item 3)

- `internal/tools/delete_workspace.go`:
  - 引数 struct に `DryRun bool` 追加
  - `dry_run=true`: `Manager.DescribeForDelete(ctx, id)` 的な新メソッドで container_state + host_paths + disk_usage_bytes を計算、`{would_delete, ...}` を返却
  - `dry_run=false`: 既存挙動 (Manager.Delete)
- `Manager` 側に `Describe(ctx, id) (*WorkspaceDeletePreview, error)` 的な readonly メソッド追加
- `disk_usage_bytes` は `filepath.Walk` で集計

#### 2d. query_data table-not-found hint (item 4)

- `internal/tools/query_data.go`:
  - 既存の `script_failed` エラー組み立て前に、stderr に `CatalogException: Table "<name>" does not exist` が含まれるか検出 (regex)
  - 該当: 同 workspace で `SHOW TABLES` を実行 (短い script、別の execScript 呼び出し)、`Manager.List(ctx)` で他 workspace 取得
  - 構造化エラーの `details` に `missing_table` / `available_tables_in_this_workspace` / `other_workspaces` を追加
  - パース失敗時は無視して既存挙動

#### 2e. describe_workspace 新ツール (item 5)

- `internal/tools/describe_workspace.go` 新規:
  ```go
  type describeWorkspaceArgs struct { WorkspaceID string `json:"workspace_id"` }
  type DescribeWorkspaceResult struct {
      WorkspaceID    string `json:"workspace_id"`
      HostWorkDir    string `json:"host_work_dir"`
      ContainerState string `json:"container_state"`
      Tables         []TableInfo `json:"tables"`
  }
  type TableInfo struct {
      Name    string `json:"name"`
      Columns []ColumnInfo `json:"columns"`
  }
  type ColumnInfo struct { Name, Type string }
  ```
- 実装: `Manager.Ensure` → Python で `SHOW TABLES` + 各 `DESCRIBE` → JSON 化
- `internal/tools/tools.go` `Register` に追加 (8 → 9)

## 4. Open questions

- **`describe_workspace` で `_upload`/`_code` のような内部テーブル名は除外するか** → DuckDB の `SHOW TABLES` には system table が含まれない、ユーザー table のみなので素直に全部返す
- **`disk_usage_bytes` の計算が大規模 workspace で遅い場合** → 1 秒タイムアウトで打ち切り、`disk_usage_bytes: null` + `disk_usage_unavailable: "computation_too_slow"` を返す方針。MVP では実装せず、必要になったら追加
- **`query_data` の `total` フィールドを常にゼロ値で送るか** (`omitempty`) → 常に送る (truncate されてないケースは row_count と同値)。LLM が `total` の有無を判定する手間を減らす
