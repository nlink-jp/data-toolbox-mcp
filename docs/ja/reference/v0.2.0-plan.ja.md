## 2. Work breakdown (5 トラック)

### Track 1: Dockerfile + matplotlibrc 拡張 (ADR-0007 実装)

- `runtime/Dockerfile`:
  ```dockerfile
  FROM python:3.12-slim

  RUN apt-get update && apt-get install -y --no-install-recommends \
          fonts-noto-cjk ca-certificates \
      && rm -rf /var/lib/apt/lists/*

  RUN pip install --no-cache-dir \
          duckdb~=1.1 pandas~=2.2 polars~=1.8 pyarrow~=18.0 \
          matplotlib~=3.10 Pillow~=11.0

  RUN mkdir -p /etc/matplotlib && \
      printf 'font.family: sans-serif\nfont.sans-serif: Noto Sans CJK JP, DejaVu Sans, Arial, Liberation Sans\naxes.unicode_minus: False\n' \
        > /etc/matplotlib/matplotlibrc
  ENV MATPLOTLIBRC=/etc/matplotlib/matplotlibrc

  RUN useradd -m -u 1000 toolbox
  USER 1000:1000
  WORKDIR /work
  CMD ["sleep", "infinity"]
  ```
- `cmd/build_runtime_test.go` の `TestEmbeddedDockerfileContents` を新 Dockerfile に合わせて更新

**DoD**:
- `dist/data-toolbox-mcp build-runtime` でイメージ生成成功
- `podman images` でサイズ確認 < 900MB
- `podman run --rm <image> python -c "import matplotlib, PIL; print(matplotlib.__version__, PIL.__version__)"` 動作
- `podman run --rm <image> python -c "import matplotlib; matplotlib.use('Agg'); import matplotlib.pyplot as plt; plt.title('日本語テスト'); plt.savefig('/tmp/test.png'); print('ok')"` 動作

### Track 2: Backend (Manager.List/Delete + Podman.ContainerState + runtime manifest)

- `internal/workspace/podman.go` に `ContainerState(ctx, name) (string, error)` 追加 (`"running"` / `"stopped"` / `"absent"`)
- `internal/workspace/manager.go` に:
  - `List(ctx) ([]WorkspaceInfo, error)` — ディスク走査 + ContainerState 取得
  - `Delete(ctx, id) error` — Validate → path traversal 二重防御 → コンテナ削除 → ディスク削除
- `internal/runtime/manifest.go` (新規) に describe_runtime が返す静的データを定数で定義:
  ```go
  type Package struct { Name, VersionConstraint string }
  type Manifest struct {
      PythonVersion    string
      ContainerImage   string  // 動的に置換可能 (config 経由)
      Packages         []Package
      Fonts            []string
      MountPoints      map[string]string
      Notes            []string
  }
  var Default = Manifest{...}
  ```
  (network は実行時に config から取得するので manifest には入れない)

**DoD**:
- 既存テスト全 pass
- `Manager.List()` / `Delete()` の unit test (fake runner) pass
- `Manifest.Default` のフィールドが Dockerfile と整合 (Track 4 のテストで検証)

## 3. Track dependencies

```
T1 (Dockerfile) ─┬─▶ T4 (tests: matplotlib smoke / manifest drift)
                 └─▶ T2 (manifest constants)
T2 (backend) ────────▶ T3 (frontend: tools depend on Manager.List/Delete)
T3 (frontend) ───────▶ T4 (E2E needs all tools registered)
T4 (tests) ──────────▶ T5 (release blocked on green tests)
```

T1 と T2 は並列着手可。T3 は T2 待ち。T4 は T1 + T2 + T3 揃ってから。T5 は最後。

## 4. Open questions

- **manifest drift テストの厳密性**: package name セット一致のみで十分か、バージョン pin の prefix 一致 (`duckdb` パッケージのインストール済バージョンが `1.1.x` で始まること) まで検証するか？ → MVP では name セットのみ、版数ドリフトは Phase 3 で TTL GC と一緒に強化検討
- **`list_workspaces` の空配列処理**: workspace_dir 自体が存在しない場合の挙動? → `[]` を返す (エラーにしない)、`os.IsNotExist` チェックで処理
- **`describe_runtime` の `notes` フィールド**: 国際化どうする？ → Phase 3 で考える、v0.2.0 は英語のみで OK
