# ADR-0010: v0.4.0 UX polish — describe_workspace + query_data 戻り値拡張 + delete_workspace dry_run + 説明文 hints

- Status: Accepted
- Date: 2026-06-06
- Driver: magi
- Generalises to: なし

---

## Context

v0.3.0 Claude Desktop 実機検証で LLM 自身が「優先度高」とした 2 項目 (`attach_files` + `load_from_work`) は実装済みで「コアの実用ループ」は閉じた。残ったのは「あれば便利」レベルの細かい摩擦 5 件:

1. ツール description が薄く、`attach_files` / `load_from_work` のような **「いつ使うべきか」** が `describe_runtime` を呼ぶまで分からない (description 起点で発見できないと、最初の数ターンで遠回りする)
2. `query_data` が暗黙の `LIMIT 20000` を効かせたとき、戻り値の `limit_reached: true` だけでは **本当の総行数が分からない** 。「あと何件あるのか」を LLM がユーザーに案内できない
3. `delete_workspace` が **不可逆かつ事前確認なし** 。LLM の workspace_id 取り違えで生きた workspace が消える可能性がある (MCP クライアントの確認 UI が唯一の防御)
4. `query_data` でテーブル不在エラー (`CatalogException: Table "sales" does not exist`) のとき、メッセージが **そのまま DuckDB 由来の素っ気ない文言** 。LLM が「テーブル名のタイポ」「workspace_id の取り違え」「未ロード」のどれかを判別する材料がない
5. セッションを跨いで「この workspace に何の table があるか」を確認するには、LLM が **手動で `information_schema` クエリを叩く** 必要がある。1 ツール `describe_workspace(workspace_id)` で済めば素直

いずれも v0.3.0 のコア機能を変えず、表面の体感を底上げする範囲。

## Decision

5 項目を 1 リリース (v0.4.0) で対応する。それぞれ個別 ADR を立てるほどの構造的決定ではないが、まとめて 1 ADR にすることで「UX polish」のテーマと変更範囲を 1 文書で把握できるようにする。

### 1. ツール description の hint 一行追加

各ツールの `inputSchema.description` を補強:

| ツール | 追加する hint |
|--------|--------------|
| `load_data` | "For files generated by execute_code that already live in /work, use load_from_work instead (no allowed_paths setup needed)." |
| `execute_code` | "Files written to /work appear at host_work_dir; return them to the user with attach_files instead of base64-encoding into stdout." |
| `query_data` | "If no LIMIT clause, an outer LIMIT (default 20000) is auto-applied; the result carries `truncated: true` and `total: N` when this trips." |
| `delete_workspace` | "Irreversible. Set dry_run: true to see what would be removed without acting." |

実装変更なし、文字列差分のみ。後方互換。

### 2. `query_data` 戻り値に `truncated` + `total` を追加

現状:
```json
{"rows": [...], "row_count": N, "limit_applied": 20000, "limit_reached": true}
```

v0.4.0:
```json
{
  "rows": [...],
  "row_count": N,
  "limit_applied": 20000,
  "limit_reached": true,    // 後方互換のため残す
  "truncated": true,        // 別名 (より一般的な terminology)
  "total": 47832            // LIMIT 無しなら実行したときの総行数 (常に設定)
}
```

- `total` は **truncate されたときだけ** 内部で追加の `SELECT COUNT(*) FROM (user_sql) sub` を実行して取得。非 truncate 時は `total = row_count`
- 余計な COUNT() のコストは「実際に切り詰まったときだけ」発生するので、典型クエリ (LIMIT 内収まる) には負荷増なし
- COUNT() 実行が timeout した場合 (元クエリは通ったが COUNT が遅い): `total: null` で返却し、`total_unavailable_reason: "count_timed_out"` を添える

### 3. `delete_workspace` に `dry_run` 引数を追加

引数:
```json
{"workspace_id": "samples", "dry_run": true}   // 既定 false
```

戻り値 (dry_run=true のとき):
```json
{
  "would_delete": true,
  "workspace_id": "samples",
  "container_id": "ab12cd34..." | null,
  "container_state": "running" | "stopped" | "absent",
  "host_paths": {
    "base": "/Users/magi/.data-toolbox/samples",
    "work": "/Users/magi/.data-toolbox/samples/work",
    "duckdb": "/Users/magi/.data-toolbox/samples/work/analysis.duckdb"
  },
  "disk_usage_bytes": 123456789
}
```

戻り値 (dry_run=false, 既存挙動):
```json
{"deleted": true, "workspace_id": "samples"}
```

- 引数追加なので **後方互換** (`dry_run` 省略時は既存動作)
- `disk_usage_bytes` はディレクトリツリーの walk 合計
- LLM が dry_run の結果をユーザーに見せて「これを消しますか？」と確認 → ユーザー OK で `dry_run: false` で再呼び出し、というフローが取りやすくなる

### 4. `query_data` のテーブル不在エラーに hint を追加

実装 flow:
1. Python script の出力エラーをパース
2. `CatalogException: Table "<name>" does not exist` のパターンを検出
3. 同じ workspace で `SHOW TABLES` を実行 → 候補テーブル取得
4. `Manager.List(ctx)` で他 workspace を取得
5. 構造化エラー `script_failed` の `details` に hint を追加:

```json
{
  "code": "script_failed",
  "message": "query_data script failed (exit 1): CatalogException: Table \"sales\" does not exist",
  "details": {
    "exit_code": 1,
    "stderr": "...",
    "missing_table": "sales",
    "available_tables_in_this_workspace": ["products", "logs"],
    "other_workspaces": ["samples", "analysis-2"]
  }
}
```

- 既存の構造化エラー経路 (`internal/toolerr`) を拡張するだけ、新規エラーコードなし
- LLM はこの details を見て「テーブル名のタイポ」「workspace_id の取り違え」「未ロード」を切り分けられる
- パース失敗時 (DuckDB のエラー文言が将来変わった場合等) は hint を添えずに今まで通りのエラーを返す (degraded gracefully)

### 5. 新ツール `describe_workspace(workspace_id)`

- **引数**: `{workspace_id: string}`
- **戻り値**:
  ```json
  {
    "workspace_id": "samples",
    "host_work_dir": "/Users/magi/.data-toolbox/samples/work",
    "container_state": "running",
    "tables": [
      {
        "name": "sales",
        "columns": [
          {"name": "order_id", "type": "BIGINT"},
          {"name": "order_date", "type": "DATE"},
          ...
        ]
      },
      {"name": "products", "columns": [...]}
    ]
  }
  ```
- **実装**: `Manager.Ensure` → Python で `SHOW TABLES` + 各テーブル `DESCRIBE` → JSON 化
- 行数 (`row_count`) は **含めない** (v0.4.0 では)。SELECT COUNT を全テーブル分実行すると重い。需要が出たら別 ADR で追加検討
- `list_workspaces` (workspace 一覧) と `describe_workspace` (1 workspace の中身) の対称形 → ツール命名で意図が明確

ツール表面: 8 → 9 (`describe_workspace` 追加)。

## Consequences

**Positive:**

- **発見性向上 (項目 1)**: 初手 `tools/list` で description を読んだ時点で `attach_files` / `load_from_work` の使い所が分かる
- **行数の安心感 (項目 2)**: `total` で「あと何件あるか」が一目、LLM がユーザーに正確な要約を返せる
- **事故防止 (項目 3)**: `dry_run` で削除前の確認シナリオが作れる、MCP クライアントの user approval gate と二重防御
- **エラー親切化 (項目 4)**: workspace_id 取り違えへの即気付き、LLM の「自分で気づいて修正する」ループが短くなる
- **introspection 完備 (項目 5)**: セッション跨ぎで「この workspace に何があるか」を 1 ツールで把握、`information_schema` の手動クエリ不要
- すべて純粋に additive、引数 / 戻り値の削除なし

**Negative:**

- ツール表面 8 → 9。LLM の認識コスト微増
- `query_data` の総行数取得で truncate 時に追加 COUNT() が走る (許容範囲、truncate という稀なケースのみ)
- `describe_workspace` 実行のたび `Ensure` + Python exec で 5-10 秒のオーバーヘッド (typical podman exec 通り)。`list_workspaces` よりは重い
- ドキュメント・テストの維持工数微増 (9 ツール分の inputSchema)

## Alternatives Considered

### A1: 5 項目を別々の ADR (ADR-0010 〜 ADR-0014) に分割

- Pros: 各 ADR の粒度が細かく、レビューもしやすい
- Cons: 5 件中 4 件が「既存ツールの拡張」で構造的決定が小さい、ADR の数だけ起票・レビューの工数が増える、リリースも刻むと総工数が増える
- 却下: 1 リリース (v0.4.0) で出すこと自体は決定済み (Option B 採択)、ADR を分けることに別途の利益が薄い

### A2: `describe_workspace` の代わりに `list_tables(workspace_id)` 単機能ツール

- Pros: 戻り値が単純、実装も軽い
- Cons: LLM はだいたい「テーブル一覧」の次に「各カラム」を知りたい、2 ツール往復になる、命名上の対称性 (`list_workspaces` vs `list_tables`) は同じ抽象階層感が出ない
- 却下: `describe_workspace` 1 発のほうが対称形 (1 workspace の中身全部) で使いやすい

### A3: `delete_workspace` の `dry_run` ではなく **常に 2 段階削除** (`prepare_delete` → `confirm_delete`) にする

- Pros: 一度確認しないと消えない、最大限の事故防止
- Cons: ツール 1 → 2 に分裂、UX が大袈裟、MCP クライアントの user approval gate と二重で煩雑
- 却下: dry_run 引数の追加で十分、強制 2 段階は LLM のスムーズな運用を阻害

### A4: `query_data` の `total` を常に取得 (毎回 COUNT)

- Pros: 戻り値の整合性が一貫
- Cons: 全クエリに COUNT のコスト、典型ケース (LIMIT 内収まる) で無駄
- 却下: truncate されたときだけ COUNT を実行する条件分岐のほうが合理的

### A5: 項目 1 (description) のみ対応、他は保留

- Pros: 工数最小、ADR 不要
- Cons: 残り 4 項目の摩擦は据え置き
- 却下: Option B 採択により全 5 項目をまとめる方針
