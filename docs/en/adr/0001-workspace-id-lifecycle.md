# ADR-0001: workspace_id scoping and lifecycle

- Status: Accepted
- Date: 2026-06-05
- Driver: magi
- Generalises to: none

---

## Context

The predecessor `shell-agent-v2` scopes its DuckDB engine (`app/internal/analysis/engine.go`, lazy-open Engine) and its Podman container (`app/internal/sandbox/cli.go`, `EnsureContainer(ctx, sessionID)`) per UI session. The `sessionID` is generated by the Wails UI layer and persists state on the host under the flat `base-dir/sessionID/{analysis.duckdb, work/, messages.jsonl}`.

When we extract these capabilities into an MCP server, the stdio MCP "session" concept becomes fragile:

- One stdio connection roughly equals one session, but Claude Desktop restarts and Cursor MCP reconnects break it
- Discarding the DuckDB file and container on every disconnect forces preprocessing to be redone — not viable for real use
- Conversely, "one container for the entire server lifetime" lets independent analytical tasks contaminate each other's state

The LLM/MCP client therefore needs an explicit way to address **"this bundle of data = this scope of state."**

## Decision

The first argument to all three tools (`load_data`, `query_data`, `execute_code`) is **`workspace_id: string`**, an explicit key supplied by the LLM/client. It scopes:

- One Podman container
- One DuckDB file
- One host-side `/work` directory

The on-disk layout follows the shell-agent-v2 flat structure:

```
<workspace_dir>/
└── <workspace_id>/
    ├── analysis.duckdb
    └── work/        # mounted to /work inside the container
```

As long as the same `workspace_id` is supplied, the container and DuckDB file are reused. After a server restart, the state is reconstructed from disk (the DuckDB file and directory persist). In-memory container references are volatile, so when a `workspace_id` is supplied we ask Podman "is this container already running?", and if not we start it (idempotent `ensure_workspace`).

`workspace_id` syntax constraints:

- `^[a-zA-Z0-9_-]{1,64}$` (path-traversal defense and container-name safety)
- Recommended to use human-readable values like `analysis-2026Q2` since the LLM reads/writes the string

## Consequences

**Positive:**

- Multiple independent analytical tasks coexist safely because state location is explicit
- Data persists across server restarts, enabling long-running analytical sessions
- The `workspace_id` namespace is just strings, so a different MCP client referring to the same name participates in the same workspace
- Pruning unused workspaces is straightforward at the `workspace_id` granularity (a human can comfortably delete one)

**Negative:**

- The LLM acquires a responsibility to "remember the state key." UX aids like accepting a default `workspace_id` of `"default"` will be needed
- An unbounded number of workspaces eventually requires garbage collection → a TTL-based cleanup will be considered in Phase 2 via a separate ADR
- Concurrent access to the same `workspace_id` (multiple clients at once) is effectively serialized by DuckDB's single-writer constraint → Phase 1 minimal stance: "concurrent access to the same workspace errors out the second request"

## Alternatives Considered

### A1: 1 MCP server = 1 container = 1 DuckDB (fixed for the lifetime)

- Pros: simplest API; `workspace_id` can be dropped from arguments
- Cons: cannot handle multiple independent analyses from a single LLM client; data contamination is easy
- Rejected because even on a personal machine, juggling multiple project analyses concurrently is normal, and data mixing is fatal

### A2: Per-MCP-session create-and-destroy (the shell-agent-v2 path)

- Pros: state management is declarative; no garbage collection needed
- Cons: DuckDB must be rebuilt every time stdio reconnects. Data is lost when Claude Desktop restarts
- Rejected because the practical friction is too high

### A3: Persistent host-side DuckDB + ephemeral container

- Pros: container restart cost stays low
- Cons: DuckDB is single-writer; writes from inside the Python container would race the host. The design becomes complex
- Rejected because it leaves A1's state-scoping problem unsolved while adding implementation cost

These alternatives are also recorded in the RFP Discussion Log § "Lifecycle model."
