# ADR-0010: v0.4.0 UX polish — describe_workspace + query_data result extensions + delete_workspace dry_run + description hints

- Status: Accepted
- Date: 2026-06-06
- Driver: magi
- Generalises to: none

---

## Context

The two items the LLM itself flagged as "highest impact" during the v0.3.0 Claude Desktop verification (`attach_files` + `load_from_work`) are shipped, closing the core practical loop. What remains is a set of "would be nice" frictions:

1. Tool descriptions are thin; **when to use `attach_files` / `load_from_work`** isn't discoverable from `tools/list` alone, forcing detours in the first few turns until `describe_runtime` is read.
2. When `query_data`'s implicit `LIMIT 20000` trips, `limit_reached: true` is set but the **actual total row count is unknown**, so the LLM can't tell the user "X more rows exist beyond what's shown."
3. `delete_workspace` is **irreversible with no pre-confirmation**. An LLM workspace_id typo could erase a live workspace; the MCP client's user-approval UI is the only defense.
4. When `query_data` hits a missing table (`CatalogException: Table "sales" does not exist`), the message is **the raw DuckDB output**, with no signal about whether it's a typo, a wrong workspace_id, or an unloaded table.
5. To know "what tables are in this workspace" across sessions, the LLM has to **manually issue `information_schema` queries**. A single tool `describe_workspace(workspace_id)` would be cleaner.

All five items leave v0.3.0's core unchanged; they raise the experience floor at the surface.

## Decision

Address all five in one release (v0.4.0). None individually justifies a separate ADR's overhead, but bundling them as one keeps the "UX polish" theme and the change surface visible in a single document.

### 1. Tool description hint additions

Augment each tool's `inputSchema.description`:

| Tool | Added hint |
|------|------------|
| `load_data` | "For files generated by execute_code that already live in /work, use load_from_work instead (no allowed_paths setup needed)." |
| `execute_code` | "Files written to /work appear at host_work_dir; return them to the user with attach_files instead of base64-encoding into stdout." |
| `query_data` | "If no LIMIT clause, an outer LIMIT (default 20000) is auto-applied; the result carries `truncated: true` and `total: N` when this trips." |
| `delete_workspace` | "Irreversible. Set dry_run: true to see what would be removed without acting." |

No implementation change, string diffs only. Backward compatible.

### 2. Add `truncated` + `total` to `query_data` result

Current:
```json
{"rows": [...], "row_count": N, "limit_applied": 20000, "limit_reached": true}
```

v0.4.0:
```json
{
  "rows": [...],
  "row_count": N,
  "limit_applied": 20000,
  "limit_reached": true,    // kept for backward compat
  "truncated": true,        // alias with more common terminology
  "total": 47832            // total row count from the un-LIMIT-ed user query (always set)
}
```

- `total` is fetched via an **additional `SELECT COUNT(*) FROM (user_sql) sub`** only when truncation occurred. Non-truncated case: `total = row_count`.
- The extra COUNT() cost lands only on actual truncation — typical queries (inside LIMIT) pay nothing extra.
- If the COUNT() itself times out (original query was OK, but COUNT is slow): return `total: null` with `total_unavailable_reason: "count_timed_out"`.

### 3. Add `dry_run` argument to `delete_workspace`

Arguments:
```json
{"workspace_id": "samples", "dry_run": true}   // default false
```

Result with dry_run=true:
```json
{
  "would_delete": true,
  "workspace_id": "samples",
  "container_id": "ab12cd34..." | null,
  "container_state": "running" | "stopped" | "absent",
  "host_paths": {
    "base": "/Users/magi/.data-toolbox/samples",
    "work": "/Users/magi/.data-toolbox/samples/work",
    "duckdb": "/Users/magi/.data-toolbox/samples/work/analysis.duckdb"
  },
  "disk_usage_bytes": 123456789
}
```

Result with dry_run=false (existing behavior):
```json
{"deleted": true, "workspace_id": "samples"}
```

- Argument is additive → **backward compatible** (omitting `dry_run` preserves existing behavior).
- `disk_usage_bytes` is computed by walking the directory tree.
- Enables a "show what would be deleted → user confirms → re-call with dry_run: false" flow.

### 4. Hint for `query_data` missing-table errors

Implementation flow:
1. Parse Python script's stderr for `CatalogException: Table "<name>" does not exist`.
2. Issue `SHOW TABLES` in the same workspace to enumerate available tables.
3. `Manager.List(ctx)` enumerates other workspaces.
4. Extend the existing structured `script_failed` error's `details`:

```json
{
  "code": "script_failed",
  "message": "query_data script failed (exit 1): CatalogException: Table \"sales\" does not exist",
  "details": {
    "exit_code": 1,
    "stderr": "...",
    "missing_table": "sales",
    "available_tables_in_this_workspace": ["products", "logs"],
    "other_workspaces": ["samples", "analysis-2"]
  }
}
```

- Stays within the existing `internal/toolerr` machinery; no new error code.
- The LLM can use `details` to distinguish "typo" / "wrong workspace_id" / "not yet loaded".
- If the parse fails (e.g. DuckDB future-changes its error text), we just return the unmodified error — graceful degradation.

### 5. New tool `describe_workspace(workspace_id)`

- **Arguments**: `{workspace_id: string}`
- **Result**:
  ```json
  {
    "workspace_id": "samples",
    "host_work_dir": "/Users/magi/.data-toolbox/samples/work",
    "container_state": "running",
    "tables": [
      {
        "name": "sales",
        "columns": [
          {"name": "order_id", "type": "BIGINT"},
          {"name": "order_date", "type": "DATE"},
          ...
        ]
      },
      {"name": "products", "columns": [...]}
    ]
  }
  ```
- **Implementation**: `Manager.Ensure` → Python `SHOW TABLES` + `DESCRIBE` for each → JSON.
- Row counts (`row_count`) are **excluded** in v0.4.0. SELECT COUNT across all tables is heavy. Add later via a separate ADR if demand surfaces.
- Symmetry: `list_workspaces` (the workspaces) ↔ `describe_workspace` (one workspace's contents). Naming makes intent obvious.

Tool surface: 8 → 9 (`describe_workspace` added).

## Consequences

**Positive:**

- **Discoverability (item 1)**: reading `tools/list` once surfaces when to reach for `attach_files` / `load_from_work`.
- **Row-count confidence (item 2)**: `total` says "X more rows exist," so the LLM can summarize honestly.
- **Mishap defense (item 3)**: `dry_run` enables a "show then confirm" pattern, layered on the MCP client's approval gate.
- **Error friendliness (item 4)**: workspace_id typos surface immediately; the LLM's self-correction loop shortens.
- **Cross-session introspection (item 5)**: "what's in this workspace?" answered by one tool, no manual `information_schema` queries.
- All purely additive; no removed arguments or fields.

**Negative:**

- Tool surface 8 → 9; LLM recognition load creeps up a notch.
- The truncation-only COUNT() in `query_data` adds latency in the truncated case (acceptable).
- `describe_workspace` runs `Ensure` + a Python exec each call, so 5–10 s overhead is typical (consistent with our other `podman exec` tools, but heavier than `list_workspaces`).
- Slight maintenance bump on docs and tests across 9 tools.

## Alternatives Considered

### A1: Split into five ADRs (ADR-0010..ADR-0014)

- Pros: finer-grained, easier to review individually
- Cons: four of five are existing-tool extensions, so the structural decisions are small; per-ADR overhead exceeds the value
- Rejected — we already committed to one release (Option B); splitting ADRs doesn't add commensurate benefit

### A2: `list_tables(workspace_id)` instead of `describe_workspace`

- Pros: simpler return, lighter implementation
- Cons: LLM typically wants table list + each column right after; two round trips; naming asymmetry with `list_workspaces`
- Rejected — `describe_workspace` (one call surfaces everything) is the better symmetry

### A3: Force two-step delete (`prepare_delete` → `confirm_delete`)

- Pros: maximum mishap defense
- Cons: 1 tool → 2 tools, heavier UX, duplicative with the MCP client's user-approval gate
- Rejected — `dry_run` argument is enough; mandatory two-step impedes normal use

### A4: Always fetch `total` (COUNT on every query)

- Pros: consistent return shape
- Cons: full-query COUNT for every call wastes work on the common (within-LIMIT) case
- Rejected — conditional COUNT (only on truncation) is the right trade

### A5: Do only item 1 (descriptions); defer the rest

- Pros: minimal effort, no ADR needed
- Cons: leaves the other four frictions in place
- Rejected — we chose Option B (all five together)
