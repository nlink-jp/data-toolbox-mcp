# v0.3.0 Development Plan

> Status: Draft
> Date: 2026-06-06
> Prerequisite ADRs: ADR-0008 (`attach_files`) / ADR-0009 (`load_from_work`)

Implements the two highest-impact items from the LLM-driven feedback collected after the v0.2.0 Claude Desktop verification: closing the "last mile of handoff." Per `feedback_adr_before_impl`, ADR-0008 and ADR-0009 are Accepted before implementation begins.

## 1. Goals (v0.3.0 completion criteria)

- 8 MCP tools: the existing 6 + new `attach_files` + `load_from_work`.
- `attach_files` returns PNG/JPG/SVG as inline image content and CSV/JSON/MD as text content. Files above the size cap are downgraded to metadata-only.
- `load_from_work` table-izes sandbox CSV/JSON/Parquet directly via `/work/<sub>` notation.
- `internal/mcpserver` grows a `RawResult` type that lets a tool return multiple content blocks (prerequisite for `attach_files`).
- Unit + integration + E2E tests all green, including path-traversal attack scenarios.
- README × 2 / AGENTS / architecture / CHANGELOG / samples reflect the new tools.
- v0.3.0 tag + 5-platform release + darwin notarize.

## 2. Work breakdown (5 tracks)

### Track 1: architecture / v0.3.0-plan docs (this document) — done

- `architecture.md` (en/ja) §3.7 (`attach_files`) + §3.8 (`load_from_work`) added.
- This plan (`v0.3.0-plan.md` en/ja) written.

### Track 2: `mcpserver.RawResult` + `attach_files` implementation

- Add a `RawResult` type to `internal/mcpserver/server.go`:
  ```go
  type ContentBlock struct {
      Type     string `json:"type"`               // "text" | "image" | "resource"
      Text     string `json:"text,omitempty"`
      Data     string `json:"data,omitempty"`     // base64 for image/resource
      MimeType string `json:"mimeType,omitempty"`
  }
  type RawResult struct {
      Content []ContentBlock `json:"content"`
      IsError bool           `json:"isError,omitempty"`
  }
  ```
- `handleToolsCall` gains a branch: if the handler returns something satisfying `interface{ rawContent() []ContentBlock }`, use `result.content` verbatim; otherwise keep the existing JSON-marshal-into-single-text behavior.
- `internal/config/config.go` gains an `AttachConfig` section:
  ```toml
  [attach]
  max_single_size_bytes = 10485760   # 10 MiB
  max_total_size_bytes  = 20971520   # 20 MiB
  ```
- `internal/tools/attach_files.go`:
  - Argument validation (workspace_id + 1–16 paths).
  - Resolve each `/work/<sub>` or `<sub>` to `<host_work_dir>/<sub>` with `filepath.Clean` + prefix check.
  - Dispatch by extension: image / text / metadata.
  - Enforce per-file and cumulative size caps; downgrade to metadata-only on overflow.
  - Assemble and return `mcpserver.RawResult`.
- `internal/tools/tools.go` `Register` adds `attach_files` (6 → 7 tools).

### Track 3: `load_from_work` + `load_data` DRY refactor

- New `internal/tools/load_helpers.go` extracts reader-pick and Python-script assembly:
  ```go
  func chooseReader(ext string) string  // moved from load_data
  func buildLoadScript(tableName, readerFn, containerPath string) string  // new
  ```
- `internal/tools/load_data.go` is rewritten to use the helpers (no behavior change).
- New `internal/tools/load_from_work.go`:
  - Argument validation (`file_path` must start with `/work/`).
  - Strip the prefix, resolve to `<host_work_dir>/<sub>`, double-guard path traversal.
  - `Ensure` the workspace, generate the same Python via `buildLoadScript`, run via `execScript`.
  - Return shape matches `load_data`.
- `internal/tools/tools.go` `Register` adds `load_from_work` (7 → 8 tools).

### Track 4: Tests (unit + integration + E2E)

- `internal/tools/attach_files_test.go`:
  - Extension dispatch (image / text / metadata).
  - Cap downgrade (per-file overflow / cumulative overflow).
  - Path-traversal rejection (`/etc/passwd`, `../escape`, via symlink).
- `internal/tools/load_from_work_test.go`:
  - Rejects missing `/work/` prefix.
  - Reader pick (csv / json / parquet).
  - Path-traversal rejection.
- `e2e/v0_3_0_test.go`:
  - `TestE2E_v030_AttachFiles_RoundTrip`: execute_code produces a PNG → attach_files returns it as image content (base64 + mimeType).
  - `TestE2E_v030_AttachFiles_OversizeFallback`: with a deliberately-small `max_single`, oversized files come back as metadata-only.
  - `TestE2E_v030_LoadFromWork_RoundTrip`: execute_code emits a polars `write_csv` → load_from_work table-izes it → query_data confirms row count.
  - `TestE2E_v030_LoadFromWork_RejectsOutsideWork`: `file_path: "/etc/passwd"` and similar are rejected.

## 4. Open questions

- **Text overflow rendering for `attach_files`**: head + tail with an ellipsis marker, or head-only? → head + tail (preserves more context) for the MVP.
- **sha256 cost in metadata-only path**: a 100 MiB file can take seconds. Skip sha256 above 100 MiB; metadata reads `"[sha256 omitted: file too large]"`.
- **Should `load_from_work` accept nested paths like `/work/subdir/data.csv`?**: yes — `filepath.Clean` + prefix check keeps it safe and the LLM can write it naturally.
