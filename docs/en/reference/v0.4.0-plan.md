# v0.4.0 Development Plan

> Status: Draft
> Date: 2026-06-06
> Prerequisite ADR: ADR-0010 (UX polish ŌĆö describe_workspace + query_data extensions + delete_workspace dry_run + table-not-found hint + description hints)

v0.3.0 closed the practical core loop (load ŌåÆ execute ŌåÆ load_from_work ŌåÆ attach_files). v0.4.0 is a UX-polish release that bundles the five "would be nice" items the LLM flagged in its v0.3.0 feedback.

## 1. Goals (v0.4.0 completion criteria)

- 9 MCP tools: existing 8 + `describe_workspace`.
- `query_data` returns `truncated` + `total` (the COUNT runs only when truncation occurred).
- `query_data` table-not-found errors carry `missing_table` / `available_tables_in_this_workspace` / `other_workspaces` in `details`.
- `delete_workspace` accepts `dry_run`; when true, nothing is removed and the to-be-deleted shape is returned.
- Four tool descriptions (`load_data` / `execute_code` / `query_data` / `delete_workspace`) gain a one-line hint.
- Unit + E2E tests green.
- README ├Ś 2 / AGENTS / architecture / CHANGELOG / samples reflect the new behavior.
- v0.4.0 tag + 5-platform release + darwin notarize.

### Track 2: implementation (5 items)

The five items are mostly file-independent; can be done in any order, including in parallel.

#### 2a. Tool description hints (item 1)

- Append one line to `Register`'s description string in `internal/tools/tools.go` for `load_data`, `execute_code`, `query_data`, `delete_workspace`.
- Backward compatible, no tests needed (string-only diff).

#### 2b. `query_data` result extensions (item 2)

- Add to `QueryDataResult`:
  ```go
  Truncated             bool   `json:"truncated"`
  Total                 *int   `json:"total"`                                       // *int for null
  TotalUnavailableReason string `json:"total_unavailable_reason,omitempty"`
  ```
- `LimitReached` stays as an alias of `Truncated` (backward compat).
- Only when truncated, issue a second `SELECT COUNT(*) FROM (user_sql) sub` via `execScript`.
- COUNT timeout flows through the existing `context.WithTimeout`; on error set `Total = nil` + `Reason = "count_timed_out"`.

#### 2c. `delete_workspace` dry_run (item 3)

- `internal/tools/delete_workspace.go`:
  - Add `DryRun bool` to the args struct.
  - `dry_run=true`: read-only path computing container_state + host_paths + disk_usage_bytes; return `{would_delete, ...}`.
  - `dry_run=false`: existing behavior (`Manager.Delete`).
- New `Manager.Describe(ctx, id) (*WorkspaceDeletePreview, error)` (read-only).
- `disk_usage_bytes` computed via `filepath.Walk`.

#### 2d. `query_data` table-not-found hint (item 4)

- `internal/tools/query_data.go`:
  - Before building the `script_failed` error, regex-detect `CatalogException: Table "<name>" does not exist` in stderr.
  - If found: run `SHOW TABLES` in the same workspace (short script via execScript), call `Manager.List(ctx)` for other workspaces.
  - Attach `missing_table` / `available_tables_in_this_workspace` / `other_workspaces` to `details`.
  - Parse-fail ŌåÆ fall through to the original error (graceful degradation).

#### 2e. `describe_workspace` new tool (item 5)

- New `internal/tools/describe_workspace.go`:
  ```go
  type describeWorkspaceArgs struct { WorkspaceID string `json:"workspace_id"` }
  type DescribeWorkspaceResult struct {
      WorkspaceID    string       `json:"workspace_id"`
      HostWorkDir    string       `json:"host_work_dir"`
      ContainerState string       `json:"container_state"`
      Tables         []TableInfo  `json:"tables"`
  }
  type TableInfo   struct { Name string; Columns []ColumnInfo }
  type ColumnInfo  struct { Name, Type string }
  ```
- Implementation: `Manager.Ensure` ŌåÆ Python `SHOW TABLES` + `DESCRIBE` per table ŌåÆ JSON.
- Register in `internal/tools/tools.go` (8 ŌåÆ 9).

## 4. Open questions

- **Should `describe_workspace` exclude internal tables like `_upload` / `_code`?** ŌĆö DuckDB's `SHOW TABLES` already excludes system tables; user tables only. Return everything verbatim.
- **What if `disk_usage_bytes` is slow on a huge workspace?** ŌĆö Cap walking at 1 second with `disk_usage_bytes: null` + `disk_usage_unavailable: "computation_too_slow"`. Not implemented in MVP; revisit on demand.
- **Should `query_data.total` always be sent, even when zero?** ŌĆö Yes, always send (no `omitempty`). LLM doesn't have to test for presence.
